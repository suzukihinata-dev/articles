# はじめに
皆さんこんにちは、学生エンジニアのHinataです。
AIとロボティクスが好きで、最近はVLA(Vision-Language-Action)モデルやTTT(Test-Time-Train)モデルについて探究しています。

今回は、SLMとDJI Telloを使った相棒ドローンを作ってみました。

## 背景
![ウクライナ戦争で使われているドローン](https://www.aljazeera.com/wp-content/uploads/2026/04/AP26100249466664-1775824721.jpg?resize=770%2C513&quality=80)
僕はウクライナ戦争に関するニュースを見たことをきっかけに、このソフトの開発を始めました。僕は軍事技術が好きでよく軍事ニュースを見るのですが、少し前に見たニュースがドローンのニュースでした。ウクライナ戦争で多くのドローンが攻撃や偵察に使われているというニュースです。
そのニュースを見た時に、「戦場と災害現場って似ているな」と感じて、災害現場で動くドローンが作れたら良いなと思い、開発を始めました。

## ドローン活用の課題 - 通信範囲
![](https://static.zenn.studio/user-upload/3fe7cbb4dd75-20260624.png)
開発を始めた時以来、僕がユースケースとして参考にしているのが石川能登地震です。

2024年1月1日石川県を地震が襲いました。この地震の特徴として、インフラの寸断によって救助隊の活動が極めて制限されたということが挙げられます。これによって物資輸送の手段が空輸や海輸に制限され、支援が大きく遅れました。

そこで、無人のドローンを大量に飛ばしたら解決できるじゃん! と思ったのですが、中々そうはいきませんでした。このケースでドローンを利用するには大きな壁があります。

Google Street Viewで地震発生数ヶ月以内にインフラが寸断されている地点を調査したら、例えば輪島へ向かう道は、輪島から直線10km~20km手前辺りで寸断していました。
ドローンを使う上ではあまり問題になる距離ではなく。DJIのFlyCart100というモデルは149kgを保持した状態で12kmほど飛ぶようです。

それより重要な問題は通信距離です。
ドローンが目視外飛行(BVLOS)で飛行できる距離は10km。山間部ではその半分ほどに制限されます。

寸断地点から輪島まで、最短10kmだとしても通信が届きません。(何とかする技術があると後からわかったのですが)

そこで僕は、通信が遮断された状況でもドローンがタスクを実行できるように、画像と言語からロボットの動作を出力するAIモデル。VLAの研究を始めました。

## VLAからSLMへ、学習ドメインの落とし穴
VLAは自然言語と視覚情報からロボットの動作を出力するAIモデルです。ロボットと聞いて僕はてっきりドローンやローバーなどいろんなロボットの行動を出力できるマルチモーダルモデルかと思っていたのですが、実は殆どのVLAはロボットハンドや直動機構など、ごく限られたアクチュエーターでしかトレーニングされていないんですよね。

![](https://jet-mfg.com/wp-content/uploads/2025/03/robot-hand-1.png)
そこで僕はVLAでドローンを操縦することを諦め、SLMを高レイヤプランナーにした操縦システムにスコープを変更しました。

## 技術構成

### 全体のパイプライン
今回作ったシステムは、音声入力からTelloへの命令送信までを次のようなパイプラインで処理しています。

```text
マイク音声
  → ffmpeg (録音・正規化)
  → whisper.cpp tiny (文字起こし)
  → Ollama上のSLM (qwen3:0.6b)
  → 制約付きJSON
  → 決定論的バリデーション (DroneCommand)
  → Tello SDK文字列
  → UDP送信 (確認後)
```

設計上一番こだわったのは、「SLMに直接ドローンを操作させない」という点です。SLMはあくまで自然言語をJSONへ変換するプランナーで、それをTelloへ送って良いかどうかは決定論的なバリデーションコードが判断します。LLMの出力をそのまま信用せず、必ずルールベースの検証を一段挟むことで、誤動作や危険な命令の実行を防いでいます。

### 音声認識: ffmpeg + whisper.cpp
音声入力はffmpegでマイクから録音し(macOSのAVFoundation経由)、16kHz・モノラル・PCM16に正規化した上でwhisper.cpp(whisper-cli)へ渡しています。モデルはmultilingualのWhisper tiny(約75MB)で、ローカルで完結する軽量な文字起こしを実現しました。

GUIモードではブラウザのマイクで録音し、発話開始から約1.4秒の無音を検出すると自動的に録音を終了して解析に進みます。録音開始時に環境音を測定して無音判定の閾値を自動調整しているので、環境ノイズの大小に多少強くなっています。

### SLM: Ollama + qwen3:0.6bによる命令プランニング
![qwen](https://upload.wikimedia.org/wikipedia/commons/thumb/6/65/Qwen_Logo.svg/3840px-Qwen_Logo.svg.png)
文字起こしされたテキストは、Ollamaでホストしたqwen3:0.6bへ渡されます。0.6Bという小さいモデルですが、「自然言語の指示を決まったJSON Schemaの命令へ変換する」というタスクに絞ることで、ローカルでも十分な精度で動かせています。

システムプロンプトでは出力形式を以下のJSON Schemaに固定しています。

```json
{"action":"takeoff|land|stop|move|rotate|unknown","direction":null,"value":null,"reason":"..."}
```

開発中に一番苦労したのが命令の取り違えでした。「離陸」と「着陸」聞き分けられなかったんですよね。意味は正反対なのに、両方とも漢字に「陸」を含むので小型モデルは時々混同します。そこでシステムプロンプトに日英対訳のキーワード集と「離陸はrise、着陸はdescendで意味が逆」という注意書きを明記しましたが、やはりモデルが小さいこともあり安定しませんでした。Ollama呼び出し時はtemperature=0、format="json"を指定して、出力を決定論的かつ確実にパースできる形に固定しています。

危険な指示への対策として、SLMに渡す前段で「emergency」「flip」「モーター停止」などの禁止語をキーワードマッチングで弾くフィルタも入れています。LLMの判断だけに頼らず、確実に拒否すべき指示は手前で止める二重の安全策です。

### 決定論的検証: DroneCommand
SLMの出力はそのままTelloへは渡さず、`DroneCommand`というdataclassを経由させています。許可されるアクションは`takeoff`/`land`/`stop`/`move`/`rotate`/`unknown`のみで、`move`は前後左右上下20〜100cm、`rotate`は時計回り・反時計回り1〜180度という具体的な範囲まで検証します。範囲外の値や未対応の方向、SLMがJSONとして壊れた出力を返した場合は、すべて`unknown`(=何もしない)に倒すフェイルセーフ設計です。

LLMは出力がブレるものなので、「安全側に倒す」判断はLLMにさせず、すべてこの決定論的なコードに担わせているのが今回の実装で最もこだわった部分です。

### Tello制御: UDPソケットによる直接制御
TelloはSDK経由でUDP(192.168.10.1:8889)に文字列コマンドを送ると、ok/エラーを返してくる仕組みです。外部の制御ライブラリには依存せず、標準ライブラリの`socket`だけで実装しました。

安全面では以下のような処理を入れています。

- 離陸前にバッテリー残量を取得し、30%未満なら離陸命令を拒否
- 接続先IPが`192.168.10.x`(Tello機体のWi-Fiネットワーク)であることを確認してから送信
- 複数リクエストが同時に来てもソケットを使い回さないよう`threading.Lock`で排他制御

### GUI: 標準ライブラリだけのローカルWebサーバー
操作用のGUIは`http.server`の`ThreadingHTTPServer`だけで実装した、外部フレームワークに依存しないシンプルなHTTPサーバーです。`/api/plan`(テキストから命令生成)、`/api/voice`(音声から命令生成)、`/api/tello/status`(接続・バッテリー確認)、`/api/tello/execute`(命令送信)という4つのAPIだけを持つ最小構成にしています。

GUIはlocalhostのみにバインドし、Telloへの実送信は必ず`confirmed: true`を明示しないと実行できないようにしています。自動配信モードを有効にした場合も、バリデーションを通過した命令だけが送信対象になり、ページを再読み込みすると必ずOFFに戻る設計です。「うっかり離陸させない」ための保険を何重にも積んでいます。

### まとめると
- 音声認識: ffmpeg + whisper.cpp (Whisper tiny、ローカル完結)
- プランナー: Ollama上のqwen3:0.6b (SLM、JSON固定出力)
- 安全検証: DroneCommandによる決定論的バリデーション + 禁止語フィルタ
- 実行: 標準ライブラリ`socket`によるUDP直接制御 + バッテリー/接続チェック
- UI: 標準ライブラリ`http.server`のみのローカルGUI

外部ライブラリを増やさず、Pythonの標準ライブラリと外部プロセス(ffmpeg, whisper.cpp, Ollama)の組み合わせだけでMVPを完結させたのが、この構成のポイントです。SLMは「自然言語→候補命令」の変換器であり、それが安全かどうかの最終判断は常に決定論的なコードが行う——この役割分担を徹底しました。
